node总结走起

总结

这一周过的一般般吧。也许刚开始起步就是那么安静的。还是那句话：不以物喜不以己悲。上升到一种状态就很好了。这一周在公司也没有特别重的开发项目。主要还是以学习为主。但是学习了一些和工作无关的东西，所以不是很好。自己动脑想想也就知道。这不是一个好的开始。一定要站在公司角度想想。这周做了什么呢？看标题啊node！！！！这周其实做了挺多东西的。主要还是做了Node + vue + mysql/mongodb，在宿舍走了一波老路node + koa 微信公众号的开发。除了技术上的，其实对时间管理还是已经慢慢开始使用了。这是一个我非常喜欢的。这样过的才有价值了兄弟。

VUE+node+mongodb/mysql

使用了VUE + vue-resource + mongodb，地址传送门,这个地址就是整个任务具体流程。我也就是走通了一遍。并没有什么特别大的问题。之前自学node.js的开发指南时候就是这样的一个类似的demo。使用的是bootrap+express+mongodb。这个小项目目的是走通和数据库的连接数据流程。了解现在常用的库等等

VUE

VUE没什么好说的都没有写vue-router可以搞点事情

node

这里使用了express框架中文文档,node博大精深。还是需要很多知识支撑的。作为一个入流的前端node的知识需要学习的。其他的一些body-parser、fs、mongoose、path的库看看官网知道用这些来做什么就好的。有些已经集成在node里了有些还是package。

experss主要还是使用了路由的功能。路由就好像控球后卫一样，之前学习时候的比喻一直生动形象。把请求分发给不同的接口去处理web服务器的请求。这些请求对应就是需要返回给前端的。其他的一些库会用就行了。

###mongodb
参照之前的博客写的mongodb的安装入门教程就能入门啦mongodb。非关系型数据库存储的就是json。引入mongodb插件最好看看mongodb的文档。连接nodeJS还是贼爽的。

###mysql
最后使用了mysql去实现一下同样的功能，其实也不是特别的炸吧。不过使用mysql的库的时候,使用连接池的方式连接比较好。因为不能调一次就开关一次数据库，有同伴公司是用连接池的方式实现的。mysql对于大型的数据保存还是比较友好的。但是安装起来还是比较蛋疼的。出现了很多小bug。主要还是需要的mysql安装要安装microsoft visual c++ 2013的32位.一直安装64位的就是不行。原来是需要32位的。好像也没别的。

node + koa 实现微信公众号

koa

koa已经支持ES7的语法格式了，使用的是async函数，但是现在微信这个项目还没有深入下去。这周继续做的是代码风格的事情
所以 这周的总结写的是代码风格的事情。

这是内网映射的一个功能。把本地的服务印刷到公网上

总结（技术和鸡汤）

技术

这一周其实也没有特别多的技术的总结。打游戏打得天昏地暗，技术倒是没有增进多少。算了吧别说那些没有用的话。今天我的总结就讲讲前端编程实践指南。这个也是别人总结的，直接上传送门吧。前端实践指南。今天分享一下css命名的方式

bem（block element modifier）

这是一种非常格式化的元素的class的命名方式,这种命名方式看起来非常的分明。什么是block 什么是element 什么是修饰词modifier。这个规则最早还和大漠老师有问到。不过还是自己去了解清楚了。是一种很规范的命名方式。接下来还会介绍另外两种。最主要的是统一。这种用法优点就是类看起来比较清晰就像下面这样：

<div class="block">
  <div class="block-head">
    <div class="block-head-navbar"></div>
    <div class="block-head-logo"></div>
  </div>
  <div class="block-body"></div>
  <div class="block-footer"></div>
</div>

而传统的一般我会这样命名

<div class="block">
  <div class="blockHead">
    <div class="narbar"></div>
    <div class="narbarLogo"></div>
  </div>
  <div class="blockBody"></div>
  <div class="blockFooter"></div>
</div>

他这种写法在写css样式表时，能够很清楚的知道控制的是哪个标签。当然缺点也是显而易见的，就是双击并不能全部选中。只能选中某一部分，如果使用下划线“_”来命名也不是特别的友好。对于爬虫等等来说。接下来介绍两个很类似的命名方式。

OOCSS （Object Oriented CSS）and SMACSS（Scalable and Modular Architecture for CSS）

这两个的思想很类似的，就是把CSS抽离出来各司其职。因为css有很多部分有的控制布局，有的控制样式。为了工作更加的高效，搞得都是模块化的思想来做。OOCSS思想是：
结构域外观分离、容器与内容分离。这就是结构的block就写结构外观skin就写外观。容器就好定位。内容就单独设置样式。

另一种SMACSS这是一种更加分离的，我个人也比较喜欢这种样式编写方式，把样式分成五个部分

  • Base（基本），定义元素的默认样式。可以把 Normalize.css 放在这里
  • Layout（布局），可以参考一下 Susy 这种基于列宽而又独立于模块的类名
  • Module（模块），定义模块时应全部使用类名选择器，不要使用元素选择器
  • State（状态），注意要与模块的修饰器区分
  • Theme（皮肤）
    分离出来以上五种，并在相应的部分注释上这五种分类，也会很清楚看出css控制的是哪个。大家开发才会更加高效。这礼拜主要也是修改样式并没有特别多的内容。下礼拜着重回顾element的那些用法。

鸡汤

其实也不是鸡汤啦。就是 有道笔记上一个礼拜都总结的东西。如果不回顾一下，就会很快的飘走啦
7.24
不要一天笑嘻嘻，给人一种很轻浮的状态，虽然很友善。但是态度就不显得有威严。当然今天找到一种适合自己的入睡的方法。就是不断的转换睡觉的状态。不能专一的盯着一个东西。还是要分散注意力的。首先先把睡觉前眼前的东西全部归类放在书本里，确定大脑已经不再乱想东西就进入下一个状态。然而这个状态是分散的。而不是集中的，不说了，说的我都想睡觉了。

7.25
对“不以物喜，不以己悲”的思考。结合状态的思想来认为，这句话并不是简单的意识上的认知。而是上升到状态。一种冷静的状态。就存在大脑里。不管做什么事情。都会伴随的一种状态。才能让自己更加冷静的去做事情。

7.26
安排好每天的事情，这其实说了很多次的。一定要好好安排自己的时间，之后几天就**的在玩游戏。并没有好好的安排好自己的时间。

7.29
目标，这个问题已经讲了很多次了。只有目标才能让你全身心的投入进去。当然游戏有目标所以才会无脑的玩那么久。这就很好的告诉自己，一定要有目标。也不应该停止在意识。而是上升到状态。并且结合上礼拜说的价值观的排列来安排好自己的时间。加油，需要一种全身心的投入。

目标

下礼拜估计工作会比较简单。但是这就是我的机会。好好学习一下java和前端链接的。

我回来了

重新上路

不知不觉大学毕业已经快一个月了，好像一切都还没开始过一样。 但是其实都已经结束了。不如不觉就进入了社会的大家庭，很多东西都是那么新鲜但是是那么黑暗。也许这就是社会吧，这就是我们人类赖以生存的环境。一个人生活的环境还是很重要的。 也许是我一开始急着上路还没有想清楚吧。只是想忘了大学的遗憾和最后的悲伤吧。不过美好的人生也才正式开始。我需要做的就是踏实的走接下来的路。不要再一股脑的浪费时间做一些效率很低的事情。现在从新架起来了博客，就是要好好记录自己以前踩得坑好好的前进。记录自己的一些心得体会，好好反省自己的过失。

价值的排列

最近做事情总是特别的感性去做。这样做事情虽然看起来很有干劲，但是其实是非常低效的。因为没有认真的去思考事情的轻重缓急。这也许和价值观有关。在今天我把我觉得事情的轻重进行排列。虽然有一些还是最近比较感性才会排的，但是从长远的角度出发，有一些还是一定要摆在前面。

  1. 身体的健康
  2. 一颗强大的心
  3. 好好把前端做好
  4. 各种复杂的感情
  5. 其他各种杂事

我觉得活着要做的事情就在上面这些了，第三点和第四点其实还是比较难排列的。但是第一第二点是我近期的目标。也是活着必须要具备的能力。可以说没有前两个后面都是假的。好好休息好好锻炼身体，一个强大的心面对外界的压力，不要太在意外界的看法，耐得住寂寞，好好实现自己的目标，不要急躁。前端，这件事情。不简单，一定要想清楚再开始行动，要不然很容易翻车。最近的例子就是教训，这么低级的失误下次请你再也不要犯了。各种感情，比如父母亲朋好友等等。最后杂事，各种游玩放松。浪荡的浪费时间。 做每件事情之前，参考价值的排序，一定要想清楚到底要不要做 ，怎么做比较好

最后

最近在用VUE和element做一个桌面端的网页。逻辑有很多小坑，但是最主要的是想清楚自己应该怎么做。并不要太浮躁的去做，以为差不多就行了。还是要思考怎么做才比较好。本来想记录一些这个礼拜做的一些东西。但是感觉都在写样式这个礼拜。也没有特别大的坑，好好把样式实现。慢慢来吧。
