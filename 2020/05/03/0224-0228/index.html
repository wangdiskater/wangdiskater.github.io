1.问题 时间问题 对于继承

时间字段保存问题：
在继承的类的时间字段，
如果是save默认不会自动把时间存进去例如如下DO：

public class FormResultDTO extends BaseDO {

@Data
public class BaseDO {
@Id
@GeneratedValue
public long id;
@JsonFormat(pattern = “yyyy-MM-dd HH:mm:ss”)
public LocalDateTime createTime;
@JsonIgnore
public LocalDateTime updateTime;
}
数据库的字段是：

create_time datetime DEFAULT CURRENT_TIMESTAMP COMMENT ‘创建时间’,
update_time timestamp NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT ‘更新时间’,

不写成继承的类是保存时候是有时间的，写成继承的类保存之后就没有时间了。需要加一个@DateCreated注解。
这时候更新需要写成参数形式update_time才会自动更新

void updateById(Long id,String formNumber);
例子：

@Inject
FormResultRepository formResultRepository;

@Get("/create")
HttpResponse create() {
    FormResultDTO formResultDTO = new FormResultDTO();
    formResultDTO.setFormName("测试");
    FormResultDTO saveobj = formResultRepository.save(formResultDTO);
    saveobj.setFormNumber("拉跨");

//更新时间不会变？
    formResultRepository.update(saveobj);  //
    return RespUtils.SUCCESS("1");
}

@Get("/update/{resultId}")
HttpResponse update(@PathVariable Long resultId) {

// 更新时间会变

    formResultRepository.updateById( resultId,"updateFormNumber");
    return RespUtils.SUCCESS("OK");

}

注意事项之前理解是有问题的，直接update其实是数据库是修改了update_time 需要在Repository上加@Repository。

2.数据库编码

utf8与utf8mb4（utf8 most bytes 4）

MySQL 5.5.3之后增加了utfmb4字符编码
支持BMP（Basic Multilingual Plane，基本多文种平面）和补充字符
最多使用四个字节存储字符，utf8mb4是utf8的超集并完全兼容utf8，能够用四个字节存储更多的字符。
标准的UTF-8字符集编码是可以使用1-4个字节去编码21位字符，这几乎包含了世界上所有能看见的语言。
MySQL里面实现的utf8最长使用3个字符，包含了大多数字符但并不是所有。例如emoji和一些不常用的汉字，如“墅”，这些需要四个字节才能编码的就不支持。

字符集、连接字符集、排序字符集
utf8mb4对应的排序字符集有utf8mb4_unicode_ci、utf8mb4_general_ci.

utf8mb4_unicode_ci和utf8mb4_general_ci的对比：

准确性：

utf8mb4_unicode_ci是基于标准的Unicode来排序和比较，能够在各种语言之间精确排序
utf8mb4_general_ci没有实现Unicode排序规则，在遇到某些特殊语言或者字符集，排序结果可能不一致。
但是，在绝大多数情况下，这些特殊字符的顺序并不需要那么精确。
性能

utf8mb4_general_ci在比较和排序的时候更快
utf8mb4_unicode_ci在特殊情况下，Unicode排序规则为了能够处理特殊字符的情况，实现了略微复杂的排序算法。
但是在绝大多数情况下发，不会发生此类复杂比较。相比选择哪一种collation，使用者更应该关心字符集与排序规则在db里需要统一。
