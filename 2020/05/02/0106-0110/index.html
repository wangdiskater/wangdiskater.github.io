Effective Java - 6.避免创建不必要的对象

static boolean isRomanNumeral(String s) {
    return s.matches("^(?=.)M*(C[MD]|D?C{0,3})"
            + "(X[CL]|L?X{0,3})(I[XV]|V?I{0,3})$");
}

转成

// Reusing expensive object for improved performance
public class RomanNumerals {
    private static final Pattern ROMAN = Pattern.compile(
            "^(?=.)M*(C[MD]|D?C{0,3})"
            + "(X[CL]|L?X{0,3})(I[XV]|V?I{0,3})$");
    static boolean isRomanNumeral(String s) {
        return ROMAN.matcher(s).matches();
    }
}

16 状态模式 （从上午到晚上 不同时间不同判断）

在状态模式（State Pattern）中，类的行为是基于它的状态改变的。这种类型的设计模式属于行为型模式。
在状态模式中，我们创建表示各种状态的对象和一个行为随着状态对象改变而改变的 context 对象。

大话数据结构

17 适配器模式

将一个类的接口转换成客户希望的另外一个接口，Adapter模式使得原本由于接口不兼容而不能一起工作的那些类可以一起工作了

18 备忘录模式 — sl大法 栈？

在不破坏封装性的前提下，捕获一个对象的内部状态，并在该对象之外保存这个状态，这样以后就可以将该对象回复到原先保存的状态。

19 组合模式 — 部分和整体 （ — 树）

将对象组合成树形结构以表示 “部分-整体”的层次结构。组合模式使得用户对单个对象和组合对象的使用具有一致性。

22 桥接模式 （适配器模式？）

合成/聚合 复用原则
将抽象部分与他的实现部分分离，使他们都可以独立的变化。

抽象类 和 接口分离
也就是聚合

23 命令模式 烧烤摊 VS 烧烤店

解决行为请求者和行为实现者的紧耦合
核心：直接和老板说要吃什么 VS 打单出来老板就知道要什么
命令模式就是可以撤销，恢复。多次修改
这好像不能用在我们的项目中:
现在都是前后端分离的项目，很多设计模式都不太适用。

24 责任链模式

使多个对象都有机会处理请求，从而避免请求的发送者和接收者之前的耦合关系，将这个对象连成一条链，并沿着这条链传递该亲球，直到有一个对象去处理他为止。
（感觉和状态模式有点像，一步一步的向下）
感觉又有点像用在switch判断里面，链式调用的思想。

25 中介者模式（中心化）

用一个中介对象来封装有一系列的对象交互，中介者使得各对象不需要现实的相互引用，从而使其耦合松散，而且可以独立的改变他们之间的交互。

26 享元模式 flyweight

共享技术有效的支持大量细粒度的对象
注意享元的内部状态和外部状态
